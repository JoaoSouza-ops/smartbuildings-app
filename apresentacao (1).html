O Problema

Hoje, o acesso depende
de um QR Code.

O app institucional gera um QR Code estático que libera catracas e portas. Não há verificação de identidade — apenas posse de uma imagem na tela.

Modelo Atual · QR Code

Frágil por design

  • Print, foto ou screenshot circulam livremente entre alunos
  • Não verifica quem está apresentando o código — só que o código existe
  • Sem granularidade: o mesmo QR vale para qualquer porta, qualquer horário
  • Sem trilha de auditoria confiável — não há prova de presença real
  • Depende do celular com bateria, internet e o app aberto
SCARF · Biometria

Identidade comprovada

  • O rosto é a credencial — não há o que compartilhar
  • Verificação 1:N contra embeddings 128D armazenados com segurança
  • Permissões granulares por porta, perfil e janela de horário
  • Log assinado de cada tentativa com X-Request-ID
  • Funciona offline na borda; aluno não precisa de celular

API · Decisões

Por que cada cabeçalho
está ali.

⊕

Idempotency-Key

Catraca pode reenviar a mesma requisição por timeout de rede. Sem idempotência, o aluno passaria duas vezes no log. Inspirado em Stripe e PayPal.

⊜

RFC 7807 Problem+JSON

Erros padronizados com type, title, status, detail, instance. Cliente trata 401 de senha igual em qualquer endpoint.

⊙

X-Request-ID

Cada requisição rastreável do firmware ao banco. Auditoria forense de quem tentou entrar onde, quando, com qual confiança.

⊗

Versionamento por URL

/v1/ explícito. Quebra de contrato exige /v2/ — clientes embarcados em ESP32 não se atualizam sozinhos.

⊞

Cursor Pagination

Listagens grandes (eventos de acesso) usam cursor opaco em vez de offset. Performance estável independente do tamanho da tabela.

⊠

OAuth2 + JWT (RS256)

Tokens assinados assimétricos. Catraca valida sem chamar o servidor de auth. Refresh tokens com rotação.

Backend · main.py

FastAPI, async,
e visão na thread certa.

O loop assíncrono nunca bloqueia. Operações CPU-bound (dlib, OpenCV) rodam em pool dedicado. pgvector faz o trabalho pesado de similaridade dentro do Postgres.

@app.post("/v1/access/verify", response_model=AccessResponse)
async def verify_access(
    image: UploadFile,
    idempotency_key: UUID = Header(..., alias="Idempotency-Key"),
    request_id: UUID = Header(default_factory=uuid4, alias="X-Request-ID"),
    db: AsyncSession = Depends(get_db),
):
    # Idempotência: mesma chave → mesma resposta
    if cached := await idempotency_store.get(idempotency_key):
        return cached

    # Visão computacional fora do event loop
    embedding = await run_in_threadpool(extract_embedding, await image.read())

    # Busca vetorial nativa no Postgres
    match = await db.scalar(
        select(Person)
        .order_by(Person.embedding.cosine_distance(embedding))
        .limit(1)
    )

    if not match or match.distance > THRESHOLD:
        raise ProblemDetail(status=401, type="auth/no-match",
                            title="Identidade não reconhecida")

    await audit_log.record(person=match, request_id=request_id)
    return AccessResponse(granted=True, person=match, confidence=1 - match.distance)
Firmware · .ino

8 dólares de hardware
com disciplina de produção.

PSRAM gerenciada à mão. Upload em chunks para nunca estourar o buffer TCP. Loop não-bloqueante para manter responsividade a outros sensores.

void captureAndUpload() {
  camera_fb_t *fb = esp_camera_fb_get();
  if (!fb) return;

  WiFiClientSecure client;
  client.setCACert(rootCA);

  if (!client.connect(API_HOST, 443)) {
    esp_camera_fb_return(fb);  // libera PSRAM determinístico
    return;
  }

  // Headers + boundary multipart
  client.printf("POST /v1/access/verify HTTP/1.1\r\n"
                "Host: %s\r\n"
                "Idempotency-Key: %s\r\n"
                "Content-Type: multipart/form-data; boundary=---FA\r\n"
                "Content-Length: %u\r\n\r\n",
                API_HOST, generateUUID().c_str(), bodyLen);

  // Upload em chunks de 1KB — evita overflow do buffer TCP
  size_t sent = 0;
  while (sent < fb->len) {
    size_t chunk = min((size_t)1024, fb->len - sent);
    client.write(fb->buf + sent, chunk);
    sent += chunk;
  }

  esp_camera_fb_return(fb);  // SEMPRE liberar — PSRAM é finita
  handleResponse(client);
}
